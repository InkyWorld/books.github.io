Danger Zone Common Mistakes

Головні Антипатерни

Як НЕ треба писати код. Помилки, які вбивають проєкти за рік.

1. Anemic Domain (Анемічна модель)

Найпоширеніший гріх. Це коли ваші класи `Order`, `User` — це просто контейнери для даних (getters/setters), а вся логіка лежить у Сервісах.

❌ Як це виглядає (ПОГАНО)

Сутність тупа. Use Case розумний і займається мікроменеджментом.

# domain/entities.py
class Order:
    self.items = []
    self.total = 0
    self.status = "new"

# application/services.py
def add_item_service(order, item):
    order.items.append(item)
    order.total += item.price # Логіка зовні!
    if order.total > 1000:    # Логіка зовні!
        order.status = "vip"

✅ Як виправити (Rich Model)

Перенесіть поведінку в дані. "Скажи, не питай" (Tell, Don't Ask).

# domain/entities.py
class Order:
    def add_item(self, item):
        self.items.append(item)
        self._recalculate_total() # Сутність сама знає, як рахувати
        self._check_vip_status()  # Сутність сама оновлює статус

2. God Repository (Божественний Репозиторій)

Коли репозиторій починає робити все: відправляти пошту, рахувати аналітику, форматувати дати.

❌ Помилка

class UserRepository:
    def get_by_id(self, id): ...
    def save(self, user): ...
    
    # ЗАЙВЕ:
    def validate_password(self, password): ...
    def send_welcome_email(self, user): ...
    def calculate_daily_report(self): ...

3. Leaking Infrastructure (Інфраструктура, що протікає)

Коли типи даних з БД або Вебу проникають у бізнес-логіку.

❌ Use Case залежить від Request

# application/use_cases.py
from flask import Request # Жах! Прив'язка до фреймворку

def create_order(request: Request):
    data = request.json # Тестувати це без Flask неможливо

❌ Domain залежить від ORM

# domain/entities.py
from django.db import models # Жах! Прив'язка до БД

class Order(models.Model):
    ...

🏁 Кінець Шляху

Це кінець карти. Тепер у вас є компас (Архітектурні принципи), зброя (Патерни) та захист (Тести).
