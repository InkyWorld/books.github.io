CQRS and Event Sourcing

Divide and conquer. Advanced patterns for scalable systems.

1. CQRS (Command Query Responsibility Segregation)

In simple CRUD systems, we use the same model (ORM) for reading and writing. But read and write requirements often contradict each other:

  • Write (Command): Needs normalization, integrity, transactionality. "Hard to write".
  • Read (Query): Needs denormalization, speed, aggregation. "Fast to read".

CQRS proposes splitting the system into two parts: Command Model (write path) and Query Model (read path).

Implementation Example:

# 1. COMMAND SIDE (Clean Architecture, Rich Model)
# Using Domain Entities, Repositories, Validations
class CreateOrderUseCase:
    def execute(self, cmd: CreateOrderCommand):
        order = Order(cmd.details) # Check invariants
        self.repo.save(order)

# 2. QUERY SIDE (Optimized for Reading)
# Ignoring Entities. Reading DTOs directly via Raw SQL.
class GetOrderQueryHandler:
    def execute(self, query: GetOrderQuery) -> OrderDTO:
        # SQL query immediately returns the needed structure (JOINs included)
        sql = "SELECT o.id, u.name FROM orders o JOIN users u ON ..."
        return self.db.execute(sql).fetchone()

2. Domain Events

An event is a fact that something has already happened. "Order paid", "User registered". Events allow reducing Coupling between modules. 

from dataclasses import dataclass
from datetime import datetime

# An event is a DTO, it is immutable. Name in past tense.
@dataclass(frozen=True)
class OrderPaid:
    order_id: str
    amount: float
    paid_at: datetime

# In Entity
class Order:
    def pay(self):
        self.status = "PAID"
        # We just record the fact, not calling email_service directly
        self.events.append(OrderPaid(self.id, self.amount, datetime.now()))

# In Infrastructure (Unit of Work)
# After transaction commit, we publish events
for event in order.events:
    event_bus.publish(event)

3. Event Sourcing (ES)

Essence: We do not store the current state of the object (a row in the DB). We store the history of all events that brought the object to this state.

Traditional Approach (State-Oriented)

User(id=1, balance=500)
â¬‡ UPDATE balance=600
User(id=1, balance=600)
History lost. Where did 100 come from?

Event Sourcing

Ev1: UserCreated(balance=0)
Ev2: MoneyDeposited(1000)
Ev3: ItemPurchased(500)
-------------------------
Current State = sum(events) = 500

How it works?

  1. Event Store: This is an Append-Only database. We never delete or change events.
  2. Replay: To get an `Order` object, we take an empty order and "apply" all events from the database to it in order.
  3. Projections: Restoring an object from 1000 events every time is slow. Therefore, we listen to events in the background and update regular SQL tables ("Read Models") for fast search.
class Order(AggregateRoot):
    # State Reconstruction (Apply)
    def apply(self, event: DomainEvent):
        if isinstance(event, OrderCreated):
            self.status = "NEW"
            self.total = 0
        elif isinstance(event, ItemAdded):
            self.total += event.price

    # Load from history
    @classmethod
    def load_from_history(cls, events: List[DomainEvent]):
        order = cls()
        for event in events:
            order.apply(event)
        return order

Pros and Cons

  • Audit Log: Rock-solid audit out of the box. You cannot "quietly" fix the database.
  • Temporal Query: "What was the user's balance last Friday?" â€” Easy, just replay events up to that date.
  • Complexity: Event Versioning (Data Migration for events is hell). Eventually Consistency (Projection lag).
