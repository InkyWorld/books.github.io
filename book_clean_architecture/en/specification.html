Specification Pattern

Encapsulation of complex business rules for validation and selection.

The Specification pattern is a way to turn a complex if condition into a separate named class. It's like a "bouncer": you have a clear list of criteria for who gets in.

Problem and Solution

âŒ Without Pattern (IF Mess)

Logic is scattered, hard to read.

# In rental service
if (user.age >= 21 and 
    user.license_type == 'B' and 
    not user.has_active_loans):
    print("Can rent")

âœ… With Pattern (Clear Name)

Rule has a name and lives separately.

# Reads like human language
rule = CanRentCar()

if rule.is_satisfied_by(user):
    print("Can rent")

1. How It Works (Validation)

We create a class with one method is_satisfied_by (is it satisfied?).

class CanRentCarSpecification:
    def is_satisfied_by(self, user: User) -> bool:
        # All complex logic lives here
        has_age = user.age >= 21
        has_license = user.license_type == 'B'
        return has_age and has_license

2. Rules Constructor (Composition)

The power of the pattern is that you can combine small rules like Lego blocks using AND, OR, NOT.

is_adult = IsAdultSpec()
has_money = HasMoneySpec()

# Create a new rule from two old ones
# "Adult AND has money"
can_buy_beer = is_adult & has_money 

if can_buy_beer.is_satisfied_by(bob):
    sell_beer(bob)

3. Search (Filtering)

Instead of writing Repository methods for every case:

  • find_users_by_age(21)
  • find_users_by_age_and_money(21, 100)
  • find_users_active_and_adult(...)

We make one universal find method that takes a Specification.

# Application Layer
# "Find me all users who can rent a car"
rule = CanRentCarSpecification()
users = repository.find(rule)
Note: For this to work with a database (SQL), we need to convert Python code user.age >= 21 into SQL WHERE age >= 21 inside the repository. This is the hardest part, so libraries are often used.
