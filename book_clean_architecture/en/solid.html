Theory

SOLID Principles in Python

Five laws of agile code. If your code is difficult to test or change, you likely violated one of them.

S

Single Responsibility Principle (SRP)

"A class should have one, and only one, reason to change."

Do not confuse this: it doesn't mean a class should have only one method. It means a class should be responsible for one business function.

‚ùå God Object (Antipattern)
class Report:
    def calculate_stats(self):
        # Calculation logic
        pass

    def generate_pdf(self):
        # PDF layout logic
        pass

    def save_to_file(self):
        # Filesystem operations
        pass
‚úÖ Separation of Concerns
class StatsCalculator:
    def calculate(self): ...

class PdfReporter:
    def render(self, data): ...

class FileSystemSaver:
    def save(self, content): ...
Beginner Mistake:

Thinking that SRP forbids adding helper methods.
Example: In a `User` class, a `get_full_name()` method is fine. But `save_to_db()` is a violation of SRP, as it mixes data with persistence (storage).

O

Open/Closed Principle (OCP)

"Software entities should be open for extension, but closed for modification."

You should be able to add a new feature (e.g., a new payment method) without changing old working code.

‚ùå Bad: If/Elif Hell

class PaymentProcessor:
    def pay(self, amount, type):
        if type == "credit_card":
            # Card logic
            pass
        elif type == "paypal":
            # PayPal logic
            pass
        # To add Bitcoin, you have to modify this file and risk breaking cards!

‚úÖ Good: Polymorphism

from abc import ABC, abstractmethod

class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount):
        pass

class CreditCard(PaymentMethod):
    def pay(self, amount): ...

class PayPal(PaymentMethod):
    def pay(self, amount): ...

# Adding Bitcoin does not touch old code:
class Bitcoin(PaymentMethod):
    def pay(self, amount): ...
L

Liskov Substitution Principle (LSP)

"Subtypes must be substitutable for their base types."

If your code expects an `Animal`, it should work correctly with both `Dog` and `Cat`. If `Dog` suddenly throws an error where `Animal` shouldn't ‚Äî that's a violation.

Classic Example: Square is not a Rectangle

In geometry, a square is a special case of a rectangle. In programming, it is not! If you change the width of a rectangle, the height remains the same. For a square ‚Äî it changes.

Real World Example: ReadOnly Repository

class Repository:
    def save(self, obj): ...

class ReadOnlyRepository(Repository):
    def save(self, obj):
        # VIOLATION of LSP!
        # Client code expects save() to work, but gets an error.
        raise NotImplementedError("Read only")
Solution: Splite hierarchy into `Readable` and `Writable`.
I

Interface Segregation Principle (ISP)

"Clients should not be forced to depend on methods they do not use."

Better many small interfaces than one huge one (Fat Interface).

‚ùå Fat Interface

class SmartDevice(ABC):
    def print(self): ...
    def scan(self): ...
    def fax(self): ...

# If I have an old printer, 
# I still have to implement 
# scan() and fax(), even if empty.

‚úÖ Segregated Interfaces

class Printer(ABC):
    def print(self): ...

class Scanner(ABC):
    def scan(self): ...

# Now we can assemble MFP via
# multiple inheritance
class MFP(Printer, Scanner): ...
D

Dependency Inversion Principle (DIP)

"Depend on abstractions, not on concretions."

This is the foundation of Clean Architecture. A Use Case should not know about Postgres. It should know about an abstract `Repository`.

# ‚ùå VIOLATION of DIP: Rigid dependency
class CreateOrderService:
    def __init__(self):
        # We nailed down a specific SQL DB implementation!
        # Now this code is impossible to test without a real DB.
        self.repo = SqlAlchemyRepository() 

# ‚úÖ COMPLIANCE with DIP: Dependency Injection
class CreateOrderService:
    # We ask for "some repository" (Abstraction)
    def __init__(self, repo: OrderRepository):
        self.repo = repo
Beginner Mistake:

Using `import` inside methods or functions to "hide" the dependency. This does not solve the problem; the dependency is still rigid, just harder to find.
