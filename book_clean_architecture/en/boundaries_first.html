Boundaries First Design

Boundaries first, then entities. The main principle of modern DDD.

Why is "Entity First" a mistake?

Many start a project by drawing a database class diagram: "Here\'s the User table, here\'s Order...".

Mistake: This leads to huge "God Models". User becomes a dump of fields for billing, delivery, authorization, and profile.

Boundaries First Approach

"Boundaries are primary". Before writing User entity code, ask:

  • What context are we in?
  • What does "User" mean FOR THIS context?

Practical Example

Context: Identity & Access

Here "User" is Login, Password, 2FA.

Context: Sales

Here "User" is Customer, BuyerHistory.

Context: Shipping

Here "User" is Recipient, Address.

Instead of one large `User` class, we create three small, independent models that are linked only by ID. This is Boundaries First.
