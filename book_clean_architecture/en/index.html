Clean Architecture in Python

How to write code you won't be afraid to open six months later.

Why do we need this?

Most Python projects start the same simple way: `app.py`, a couple of SQLAlchemy models, and handler functions (views/handlers). This works while the project has 1000 lines of code. But when there are 50,000 â€” hell begins.

The "Big Ball of Mud" Problem:

  • Business logic is spread across API handlers and database models.
  • Changing the DB schema breaks API contracts.
  • Impossible to test "business" without spinning up a Docker container with Postgres.
  • Migration from Flask to FastAPI or adding GraphQL requires a complete rewrite.

Clean Architecture solves this through Dependency Inversion. Business logic becomes the center of the universe, while Framework and Database are just plugins.

The Golden Rule: The Dependency Rule

"Source code dependencies can only point INWARD, toward high-level policies."

# Pure Python, no imports!
class UserRepository(Protocol):
    def save(self, user: User): ...

def create_user(repo: UserRepository, ...):
    repo.save(...)

Details (DB) will depend on the Domain.

Architecture Layers

1

Domain (Entities)

The central circle. Contains enterprise business rules. The purest objects. Dependencies: None.

2

Application (Use Cases)

Use case scenarios. Orchestrates data flows. Tells the domain "do the work". Defines interfaces (ports) for repositories. Dependencies: Domain.

3

Adapters (Infrastructure)

Converters and implementations. SQLAlchemy repositories, API clients. Implement interfaces defined in Application. Dependencies: Application, Domain.

4

External Interfaces (Presentation)

Frameworks. FastAPI, Django views, CLI, Celery tasks. The outermost, "dirty" layer. Dependencies: All inner layers.
