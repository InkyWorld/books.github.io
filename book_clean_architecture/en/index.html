Clean Architecture in Python

How to write code you won't be afraid to open in six months.

Why do we need this?

Most Python projects start the same simple way: `app.py`, a couple of SQLAlchemy models, and handler functions (views/handlers). This works until the project reaches 1,000 lines of code. But when it hits 50,000 â€” chaos begins.

The "Big Ball of Mud" Problem:

  • Business logic is smeared across API handlers and database models.
  • Changing the database schema breaks API contracts.
  • Testing "business logic" requires spinning up a docker container with Postgres.
  • Migrating from Flask to FastAPI or adding GraphQL requires a complete rewrite.

Clean Architecture solves this through Dependency Inversion. Business logic becomes the center of the universe, while the Framework and Database are just plugins.

The Golden Rule: The Dependency Rule

"Source code dependencies can point only INWARD, toward higher-level policies."

Dependencies point inward â€” outer layers (UI, DB, External APIs, Background Jobs) depend on Domain.

Business Logic (Domain) â€” is the core that knows nothing about the external world (databases, APIs, or UI).

Layer separation: It's clear that UI and Database are at the same level of "implementation details". This emphasizes that changing the database or switching from REST to GraphQL should not affect business rules.

âŒ Wrong

# Business Logic (Domain)
# Import DB directly
from sqlalchemy.orm import Session
from myapp.models import UserModel

def create_user(db: Session, ...):
    return db.query(UserModel).add(...)

Domain depends on details (ORM).

âœ… Correct

# Business Logic (Domain)
# Pure Python, no imports!
class UserRepository(Protocol):
    def save(self, user: User): ...

def create_user(repo: UserRepository, ...):
    repo.save(...)

Details (DB) will depend on Domain.

Architecture Layers

1

Domain (Entities)

This is the "heart" of the application â€” business logic and rules that describe how your product works. Think of the domain as a set of concepts and rules: order, user, account, invariants â€” things that don't change when you swap databases or frameworks.

Why it matters: the domain should not know about implementation details (DB, HTTP, queues). Such code is easy to test (unit tests without external dependencies) and maintain long-term.

@dataclass
class Order:
    id: str
    items: List[Item]

    def add_item(self, item: Item):
        if self._closed:
            raise ValueError("Order closed")
        self.items.append(item)
2

Application (Use Cases)

This is the "orchestration" layer: it coordinates calls to the domain and adapters, executing specific use cases â€” for example, user registration or order payment.

What's in it: simple services/functions that: 1) accept input data (DTO), 2) use repositories/ports to load and save entities, 3) call domain methods. Interfaces (ports) are defined here, not their implementations.

def register_user(repo: UserRepository, dto: RegisterDTO):
    """Use case: register new user with email confirmation and referral code processing."""
    # 1. Validate input data (DTO)
    dto.validate()

    # 2. Create domain object (without infrastructure dependencies)
    user = User(email=dto.email, name=dto.name)

    # 3. Apply business rule: if there's a referral code â€” apply bonus
    if dto.referral_code:
        user.apply_referral(dto.referral_code)

    # 4. Save through port (repository interface) â€” implementation in infrastructure
    repo.save(user)

    # 5. Generate domain event for async side effects (email, metrics)
    user.record_event(UserRegistered(user.id, email=user.email))

    return user

Line-by-line explanation

  1. dto.validate() â€” validate input data: email format, required fields, and simple application-side validation.
  2. User(...) â€” create a pure domain entity; it knows nothing about DB or network clients.
  3. if dto.referral_code: user.apply_referral(...) â€” example of a business rule that changes entity state (e.g., applies bonuses).
  4. repo.save(user) â€” save through repository interface; concrete implementation (ORM, HTTP client) lives in Infrastructure.
  5. user.record_event(...) â€” add domain event; infrastructure or UnitOfWork will then send email/notifications asynchronously.
  6. return user â€” return use case result; Presentation layer will form a response for the user.

Why this is not CRUD: there are real business rules here (validation, referral logic, domain events). Use case describes not just saving a record to a table, but a scenario with mandatory steps and side effects.

3

Adapters / Infrastructure

This is where concrete implementations live: repositories for DB, external API clients, message brokers, etc. These components implement interfaces described in application and depend on domain and application.

Example: a class that uses SQLAlchemy to save a User object is an adapter; it's easy to replace with another (e.g., different ORM or external service) without changing the domain.

class SqlAlchemyUserRepo(UserRepository):
    def __init__(self, session):
        self.session = session
    def save(self, user: User):
        # map domain -> orm and persist
        self.session.add(map_to_model(user))
        self.session.commit()
4

External Interfaces (Presentation)

External shell: REST/GraphQL API, Web UI, CLI, queue tasks. Their job is to transform requests into data understandable by application and return a response to the user or system.

Important: presentation contains no business logic â€” only transformations, validation, and use case invocation.

@router.post('/users')
def create_user(body: CreateUserDTO):
    user = register_user(repo=container.user_repo, dto=body)
    return {'id': user.id}
In brief:
  • Domain â€” pure business logic (test here without external dependencies).
  • Application â€” use cases and ports (interfaces).
  • Adapters â€” concrete implementations (DB, API, brokers).
  • Presentation â€” entry points (HTTP, CLI, Background jobs).
