Config & Observability

Structured Logging, Request ID, and proper configs.

A system without logs is a black box. A system with poor logs is a mess. And hardcoded settings are a path to password leaks.

1. Observability: Request ID

Problem: a request goes through layers Controller -> Service -> Repository. If an error occurs deep in the stack, how do you know which exact HTTP request caused it?

Solution: Assign each request a unique identifier (trace_id or request_id) at the entry point and include it in every log line.

Why ContextVar here?

To avoid passing request_id as an argument to every function (e.g., get_user(user_id, request_id)), we use ContextVars. This is a Python mechanism for storing data that is globally accessible but only within the scope of the current async task (request).

Implementation (Context Vars + Structlog)

import uuid
import structlog
from contextvars import ContextVar
from fastapi import Request

# 1. Global context variable
# This will store the ID of a specific request
request_id_ctx = ContextVar("request_id", default="unknown")

# 2. Processor for Structlog (or standard logging)
# This function automatically adds request_id to all logs
def add_request_id(logger, method_name, event_dict):
    event_dict["request_id"] = request_id_ctx.get()
    return event_dict

# 3. Middleware (FastAPI)
# Wraps each request
async def logging_middleware(request: Request, call_next):
    # If ID came from a load balancer (Nginx), use it. Otherwise, generate a new one.
    req_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
    
    # ğŸŸ¢ SET: Write ID to context.
    # Now any code inside this `await call_next` can "see" this ID through `request_id_ctx.get()`
    token = request_id_ctx.set(req_id) 
    
    try:
        response = await call_next(request)
        # (Useful) Return ID to client so they can report it to support if there's an error
        response.headers["X-Request-ID"] = req_id
        return response
    finally:
        # ğŸ”´ RESET: Clear context to avoid polluting memory or other requests (though for ContextVar this is less critical than for thread-local)
        request_id_ctx.reset(token)

Result: you simply write logger.info("Something happened") in the Repository, and in the logs you see: {"event": "Something happened", "request_id": "abc-123-..."}.

2. Configuration (Pydantic Settings)

No os.getenv('DB_URL') in the middle of your code! Use typed config.

from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    DB_URL: str
    REDIS_HOST: str = "localhost"
    DEBUG: bool = False
    
    class Config:
        env_file = ".env"

# Config singleton
settings = Settings()

Settings Caching (Performance)

Creating a Settings object can be expensive (reading the .env file, parsing environment variables). If you use dependency injection (e.g., Depends in FastAPI), it makes sense to cache the result to avoid re-reading the config on every request.

from functools import lru_cache

@lru_cache
def get_settings():
    # This function will execute only once on first call.
    # On subsequent calls, the same object from memory will be returned.
    return Settings()

# Example usage in FastAPI
# from fastapi import Depends
# 
# @app.get("/info")
