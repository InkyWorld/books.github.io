Background Tasks

Celery, Workers, and Async Tasks in Clean Architecture.

In Clean Architecture, the Domain should not know about Celery, Redis, or RabbitMQ. But how do you send an email or process a video?

1. The Problem: Importing Celery in Use Case

from worker import celery_app inside a Use Case is a violation of the Dependency Rule. Your business code now depends on the task queue.

2. Solution: Task Dispatcher

1. Interface (Application Layer): Declare that we can send tasks in general.

# application/interfaces/tasks.py
from typing import Protocol, Any, Dict

class ITaskDispatcher(Protocol):
    def dispatch(self, task_name: str, payload: Dict[str, Any]):
        ...

2. Implementation (Infrastructure Layer): Celery lives here.

# infrastructure/tasks/celery_adapter.py
from celery import Celery
from application.interfaces.tasks import ITaskDispatcher

celery_app = Celery("app", broker="redis://localhost")

class CeleryTaskDispatcher(ITaskDispatcher):
    def dispatch(self, task_name: str, payload: Dict[str, Any]):
        # Send task by name (loose coupling)
        celery_app.send_task(task_name, kwargs=payload)

3. Usage in Use Case:

class RegisterUserUseCase:
    def __init__(self, tasks: ITaskDispatcher):
        self.tasks = tasks

    def execute(self, dto):
        user = User.create(dto)
        self.repo.save(user)
        
        # We just say "Send email", not knowing HOW
        self.tasks.dispatch(
            "send_welcome_email", 
            {"email": user.email, "name": user.name}
        )

3. Transactional Outbox

What if the database transaction rolls back, but the Celery task has already been sent? The user receives a "You are registered" email, but they don't exist in the database.

See the Outbox Pattern chapter for the solution (saving the task to the same DB in one transaction).
