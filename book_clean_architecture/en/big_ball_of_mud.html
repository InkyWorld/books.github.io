Big Ball of Mud

Antipattern or reality? How not to drown in the swamp.

What is it?

Definition: "A chaotically designed, sprawling system where no one knows exactly how everything is interconnected. Spaghetti code at architectural scale."

In DDD, "Big Ball of Mud" is not a swear word, but an acknowledgment of fact. Sometimes you inherit legacy that\'s impossible to refactor quickly.

What to do with it? (DDD Strategy)

  1. Don\'t try to "fix everything at once". This will kill the project.
  2. Draw boundaries (Draw a line). Acknowledge: "Inside this old module is chaos. We know this".
  3. Isolation. Build new features OUTSIDE this swamp, communicating through a strict ACL (Anti-Corruption Layer).

Real-life Example

You have a PHP 5.3 monolith with global variables. Business asks for a new loyalty program.

Bad path: Embed loyalty inside the monolith because "it has order data".

DDD path: Create a "Loyalty" microservice (clean code). Set up event synchronization from the monolith. Put a fence around the dirt.
