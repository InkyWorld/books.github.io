Layer 2 Orchestration / Logic Flow

Application Layer

This layer answers the question "WHAT does the application do?". It takes data from outside, commands entities to execute business logic, and saves the result.

Layer Anatomy

‚úÖ MUST HAVE

  • Action
    Use Cases (Interactors): Classes implementing one specific scenario (e.g., `CreateOrderUseCase`).
  • Data
    DTO (Data Transfer Objects): Simple data structures for input (Commands) and output.
  • Contracts
    Ports (Interfaces): Abstractions for communicating with the outside world (Repository Interface, EmailService Interface).
  • Logic
    Orchestration: Load user -> Check permissions -> Create order -> Save.

‚õî FORBIDDEN

  • Rules
    Complex Business Logic: Discount calculations, status checks, invariants ‚Äî all this should be inside Entities, not here!
  • Tech
    SQL / ORM / HTTP: Use Case should not know where data came from (REST, CLI, Kafka) or where it is stored (Postgres, Mongo).
  • Web
    Web Frameworks: No `flask.request` or `fastapi.Depends` inside Use Case.

1. DTO & Commands

DTO (Data Transfer Object) is a box for transferring data. It has no behavior.

The outside world (Web API) sends us JSON (primitives: strings, numbers). The Domain works with complex objects (Money, Email). The Application Layer must accept primitives via DTO and turn them into Domain objects.

Command vs Query:
  • Command: DTO that changes something (`CreateOrderDTO`).
  • Query: DTO for retrieving data (`GetOrderDTO`).
# application/dto.py
from dataclasses import dataclass
from typing import List

# Input DTO (Command)
@dataclass(frozen=True)
class CreateOrderCommand:
    customer_email: str
    items: List[dict] # [{'name': 'Book', 'price': 10.5}]

    def __post_init__(self):
        # Simple validation of input data (not business rules!)
        if not self.customer_email:
            raise ValueError("Email is required")
        if not self.items:
            raise ValueError("Item list cannot be empty")

# Output DTO (Result)
@dataclass(frozen=True)
class OrderResultDTO:
    order_id: str
    total_amount: float
    status: str

2. Ports (Interfaces)

Dependency Inversion

Use Case wants to save an order. But it cannot call `SQLAlchemy`. So it declares a Port (Interface): "I need someone who knows how to save orders".

Later, in the Infrastructure layer, we implement this interface. This allows the Use Case to depend on abstraction, not details.

# application/interfaces.py
from abc import ABC, abstractmethod
from typing import Optional
import uuid

# We import Entity because the Port works with the Domain
from domain.entities import Order 

class OrderRepository(ABC):
    """Abstract repository class. No SQL here!"""
    
    @abstractmethod
    def save(self, order: Order) -> None:
        pass

    @abstractmethod
    def get_by_id(self, oid: uuid.UUID) -> Optional[Order]:
        pass

class EmailService(ABC):
    @abstractmethod
    def send_order_confirmation(self, email: str, order_id: str):
        pass

3. Use Cases (Scenario Implementation)

This is the most important code in this layer. It ties everything together.

Use Case Algorithm:

  1. Accept DTO (Command).
  2. Convert primitives to Value Objects (str -> Email).
  3. Load/Create Entity (or several).
  4. Call Entity methods (all business logic happens THERE).
  5. Save Entity via Repository.
  6. Return result (Output DTO).
# application/use_cases.py
import uuid
from domain.entities import Order
from domain.value_objects import Money, Email
from .interfaces import OrderRepository, EmailService
from .dto import CreateOrderCommand, OrderResultDTO

class CreateOrderUseCase:
    # DEPENDENCY INJECTION:
    # We ask for any implementation of these interfaces.
    def __init__(
        self, 
        repo: OrderRepository, 
        email_service: EmailService
    ):
        self.repo = repo
        self.email_service = email_service

    def execute(self, command: CreateOrderCommand) -> OrderResultDTO:
        # 1. Prepare Value Objects
        # If email is invalid, error happens HERE (invariant protection)
        email_vo = Email(command.customer_email)
        order_id = uuid.uuid4()

        # 2. Create Entity
        order = Order(order_id, email_vo)

        # 3. Execute business actions
        for item in command.items:
            price_vo = Money(item['price'])
            # All logic "can we add item" is inside order.add_item()
            order.add_item(item['name'], price_vo)

        # 4. Save via Port
        self.repo.save(order)

        # 5. Side effects (notifications)
        self.email_service.send_order_confirmation(
            email=email_vo.value, 
            order_id=str(order.id)
        )

        # 6. Return result (Primitives only!)
        return OrderResultDTO(
            order_id=str(order.id),
            total_amount=order.total_price.amount,
            status="created"
        )

4. Multi-Entity Orchestration

Often two entities need to interact. For example: "When creating an order, if the User is VIP, apply a discount".

Who does this? The Use Case. It is the conductor.

# application/use_cases.py (extended example)

def execute(self, command: CreateOrderCommand):
    # 1. Load user via repository
    user = self.user_repo.get_by_email(command.email)
    
    # 2. Create order
    order = Order(...)
    
    # 3. Orchestration: interaction logic
    # We take data from one entity and pass it to another
    if user.is_vip():
        order.apply_discount(percent=10)
    
    # 4. Save
    self.order_repo.save(order)
Important: The Use Case must not calculate the discount itself (`total * 0.9`). It must call `order.apply_discount()`. The logic must stay in the Entities!

‚ö†Ô∏è Application Layer Antipatterns

1. Fat Use Case

This is when a Use Case starts doing calculations instead of entities.

# ‚ùå BAD
total = 0
for item in items:
    total += item.price # Calculation logic leaked from Domain!
order.total = total

# ‚úÖ GOOD
order.calculate_total() # Use Case simply delegates
2. Leaking Infrastructure

Passing framework or DB objects into a Use Case.

# ‚ùå BAD
def execute(self, request: starlette.requests.Request): # Bound to Web!
    pass

# ‚ùå BAD
def execute(self, session: sqlalchemy.Session): # Bound to DB!
    pass
3. Returning Entities

A Use Case must not return `Order` objects to the outside (Web). It must return `OrderResultDTO`.

Why? Because an Entity has behavior, while API just needs to return JSON. Plus, if you change the Entity, you break the API contract.
