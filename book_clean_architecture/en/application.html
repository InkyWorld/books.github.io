Layer 2 Orchestration / Logic Flow

Application Layer

This layer answers the question "WHAT does the application do?". It takes data from the outside, commands entities to execute business logic, and saves the result.

Layer Anatomy

âœ… MUST HAVE

What should be here
Action
Use Cases (Interactors): Classes implementing one specific scenario (e.g., CreateOrderUseCase).
Data
DTO (Data Transfer Objects): Simple data structures for input (Commands) and output.
Contracts
Ports (Interfaces): Abstractions for communication with the external world (Repository Interface, EmailService Interface).
Logic
Orchestration: Load user -> Check permissions -> Create order -> Save.

â›” FORBIDDEN

What to avoid
Rules
Complex business logic: Calculations and invariants belong inside Entities, not here.
Tech
SQL / ORM / HTTP: Using infrastructure details is forbidden.
Web
Web Frameworks: No flask.request or fastapi.Depends inside Use Cases.

1. DTO & Commands

DTO (Data Transfer Object) is a box for transferring data. It has no behavior.

The external world (Web API) sends us JSON (primitives: strings, numbers). The Domain works with complex objects (Money, Email). Application Layer must accept primitives through DTOs and convert them into Domain objects.

Command vs Query:
  • Command: A DTO that changes something (`CreateOrderDTO`).
  • Query: A DTO for retrieving data (`GetOrderDTO`).
# application/dto.py
from dataclasses import dataclass
from typing import List

# Input DTO (Command)
@dataclass(frozen=True)
class CreateOrderCommand:
    customer_email: str
    items: List[dict] # [{'name': 'Book', 'price': 10.5}]

    def __post_init__(self):
        # Simple input validation (not business rules!)
        if not self.customer_email:
            raise ValueError("Email is required")
        if not self.items:
            raise ValueError("Items list cannot be empty")

# Output DTO (Result)
@dataclass(frozen=True)
class OrderResultDTO:
    order_id: str
    total_amount: float
    status: str
Query DTO in practice:

Query doesn't change anything â€” it just reads data and returns it in a convenient form.

# application/dto.py (Query part)
@dataclass(frozen=True)
class GetOrderQuery:
    order_id: str

# Output DTO for Query response
@dataclass(frozen=True)
class OrderDetailsDTO:
    order_id: str
    customer_email: str
    items: List[dict]
    total_amount: float

Query Use Case receives `GetOrderQuery`, reads entity through a port, and returns `OrderDetailsDTO`, without changing state.

2. Ports (Interfaces)

Dependency Inversion

Use Case wants to save an order. But it cannot call `SQLAlchemy`. Therefore, it declares a Port (Interface): "I need someone who knows how to save orders".

Later, in the Infrastructure layer, we implement this interface. This allows Use Case to depend on abstraction, not on details.

# application/interfaces.py
from abc import ABC, abstractmethod
from typing import Optional
import uuid

# We import Entity because Port works with Domain
from domain.entities import Order 

class OrderRepository(ABC):
    """Abstract repository class. No SQL here!"""
    
    @abstractmethod
    def save(self, order: Order) -> None:
        pass

    @abstractmethod
    def get_by_id(self, oid: uuid.UUID) -> Optional[Order]:
        pass

class EmailService(ABC):
    @abstractmethod
    def send_order_confirmation(self, email: str, order_id: str):
        pass

3. Use Cases (Scenario Implementation)

This is the most important code in this layer. It brings everything together.

Use Case Algorithm:

  1. Accept DTO (Command).
  2. Convert primitives to Value Objects (str -> Email).
  3. Load/Create Entity (or multiple).
  4. Call Entity methods (all business logic executes THERE).
  5. Save Entity through Repository.
  6. Return result (Output DTO).
# application/use_cases.py
import uuid
from domain.entities import Order
from domain.value_objects import Money, Email
from .interfaces import OrderRepository, EmailService
from .dto import CreateOrderCommand, OrderResultDTO

class CreateOrderUseCase:
    # DEPENDENCY INJECTION:
    # We ask for any implementations of these interfaces.
    def __init__(
        self, 
        repo: OrderRepository, 
        email_service: EmailService
    ):
        self.repo = repo
        self.email_service = email_service

    def execute(self, command: CreateOrderCommand) -> OrderResultDTO:
        # 1. Prepare Value Objects
        # If email is invalid, error will be raised HERE (invariant protection)
        email_vo = Email(command.customer_email)
        order_id = uuid.uuid4()

        # 2. Create Entity
        order = Order(order_id, email_vo)

        # 3. Execute business actions
        for item in command.items:
            price_vo = Money(item['price'])
            # All logic "can we add item" is inside order.add_item()
            order.add_item(item['name'], price_vo)

        # 4. Save through Port
        self.repo.save(order)

        # 5. Side effects (notifications)
        self.email_service.send_order_confirmation(
            email=email_vo.value, 
            order_id=str(order.id)
        )

        # 6. Return result (Primitives only!)
        return OrderResultDTO(
            order_id=str(order.id),
            total_amount=order.total_price.amount,
            status="created"
        )

4. Orchestration of Multiple Entities

Often two entities need to interact. For example: "When creating an order, if the User has VIP status, apply a discount".

Who does this? Use Case. It's the conductor.

# application/use_cases.py (extended example)

def execute(self, command: CreateOrderCommand):
    # 1. Load user through repository
    user = self.user_repo.get_by_email(command.email)
    
    # 2. Create order
    order = Order(...)
    
    # 3. Orchestration: interaction logic
    # We take data from one entity and pass to another
    if user.is_vip():
        order.apply_discount(percent=10)
    
    # 4. Save
    self.order_repo.save(order)
Important: Use Case should not calculate discount itself (`total * 0.9`). It should call `order.apply_discount()` method. Logic must stay in entities!

2. Leaking Infrastructure

Passing framework or database objects to Use Case.

# âŒ BAD

    pass

# âŒ BAD
def execute(self, session: sqlalchemy.Session): # Coupling to DB!

3. Returning Entities

Use Case should not return `Order` objects to the outside (to Web). It should return `OrderResultDTO`.

Why? Because Entity has behavior, and API just needs to return JSON. Plus, if you change Entity, you'll break the API contract.
