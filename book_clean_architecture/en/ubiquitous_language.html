Ubiquitous Language

The foundation of DDD. A language spoken by both developers and business.

The Babel Tower Problem

In a typical project, a game of telephone occurs:

  • ğŸ¤µ Business: "When a customer cancels a booking, we refund their money and send a notification."
  • ğŸ‘¨â€ğŸ’» Developer (in their head): "Got it, update table bookings set status=0, insert into refunds, call mailer service."
  • ğŸ’» Code: user.disable_booking(), refund_manager.process_void()

Terms don't match. "Cancel", "Refund", "Notification" transform into "Disable", "ProcessVoid", "Mailer".

Solution: Ubiquitous Language

A programmer and an expert should speak the same language. If an expert says "Policy", there should be a Policy class in the code. If they say "Issue", there should be an issue() method.

How Does It Look in Code?

Rules

  • Use the terms in class names, method names, and module names.
  • Change the code if the business understanding changes.
  • The language is valid only within one Bounded Context.
