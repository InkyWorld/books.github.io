Policies (Strategy Pattern)

Strategy pattern in DDD. Encapsulation of complex, changing rules.

Why Do We Need This?

Sometimes business logic becomes too complex to keep it inside an Entity. Or, more often, rules change more frequently than the structure of the entity itself.

For example, calculating order total (`Order.calculate_total`). Discounts can depend on: holidays, VIP status, day of the week, purchase history. If you stuff all these `if` statements into the `Order` class, it will become a monster.

Solution: extract the algorithm into a separate class-function â€” Policy. In OOP this is known as the Strategy pattern.

Example: Calculating Discounts

1. Policy Interface (Protocol)

from typing import Protocol, List

class PricingPolicy(Protocol):
    def calculate_discount(self, order_items: List[Item], user: User) -> Money:
        ...

2. Concrete Implementations

class BlackFridayPolicy:
    def calculate_discount(self, order_items, user) -> Money:
        # Complex logic for Black Friday
        return sum(item.price for item in order_items) * 0.50

class StandardVipPolicy:
    def calculate_discount(self, order_items, user) -> Money:
        if user.is_vip:
            return sum(item.price for item in order_items) * 0.10
        return Money(0)

3. Usage in an Entity or Service

@dataclass
class Order:
    items: List[Item]
    
    # We pass the Policy as a method argument (Method Injection)
    def calculate_final_price(self, user: User, policy: PricingPolicy):
        base_price = sum(i.price for i in self.items)
        discount = policy.calculate_discount(self.items, user)
        
        self.total_price = base_price - discount
