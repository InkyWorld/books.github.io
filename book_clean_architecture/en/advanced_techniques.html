Advanced Techniques

Mapping, filtering, and async pitfalls.

When you move from theory to practice you face a lot of routine: endless data transformations, complex filters, and async pitfalls. In this chapter we explore pragmatic advanced techniques.

1. The "DTO Explosion" Problem

Architecture asks for separating DTOs per layer. In practice, for a single User entity you may end up with:

  • CreateUserRequest (Presentation, Pydantic)
  • CreateUserDTO (Application, Dataclass)
  • UserEntity (Domain)
  • UserResponse (Presentation, Pydantic)

This is tedious and demotivating. Let's consider a pragmatic approach.

Duck Typing & Protocols

Just because the Application layer shouldn't depend on Pydantic doesn't mean you can't pass Pydantic objects when they behave as required. Use Protocols.

# application/protocols.py
from typing import Protocol

class UserInput(Protocol):
    username: str
    email: str

# application/use_cases.py
def register_user(data: UserInput):
    # The Use Case is unaware of Pydantic but can work with it
    print(data.username) 

# presentation/api.py
class CreateUserRequest(BaseModel): # Pydantic model
    username: str
    email: EmailStr

# Works thanks to duck typing
register_user(CreateUserRequest(username="alex", email="a@b.com"))

Automapping (Pragmatic)

For simple CRUD operations you can use model_validate or helper methods to avoid writing field-by-field mapping.

# If fields match 1-to-1
def to_domain(dto: CreateUserRequest) -> User:
    # Risky for complex logic but OK for simple CRUD
    return User(**dto.model_dump())

2. Pagination & Filtering

Don't pass pagination parameters as function arguments. This violates the Open-Closed Principle (adding sorting would require changing the signature).

Generic Page[T]

Use a Generic DTO for all list responses.

from typing import TypeVar, Generic, List
from dataclasses import dataclass

T = TypeVar("T")

@dataclass
class PaginationParams:
    page: int = 1
    limit: int = 20
    sort_by: str = "created_at"
    desc: bool = True

@dataclass
class Page(Generic[T]):
    items: List[T]
    total: int
    page: int
    size: int
    pages: int

Repository Implementation

# infrastructure/repo.py
async def search(self, filters: OrderFilter, pagination: PaginationParams) -> Page[Order]:
    query = select(OrderModel)
    
    # 1. Apply Filters
    if filters.status:
        query = query.filter_by(status=filters.status)

    # 2. Count Total (Advanced: use separate count query)
    # total = ... 

    # 3. Apply Pagination
    query = query.limit(pagination.limit).offset((pagination.page - 1) * pagination.limit)
    
    # 4. Result
    result = await self.session.execute(query)
    models = result.scalars().all()
    
    return Page(
        items=[self._mapper.to_domain(m) for m in models],
        total=100, # placeholder
        page=pagination.page,
        size=pagination.limit,
        pages=5
    )

3. Async Traps (The Async Trap)

⚠️ Lazy Loading does not work

Attempting to access order.items in async code without preloading will raise MissingGreenlet. Use Selectinload.

# Correct way to load relations in Async
stmt = (
    select(OrderModel)
    .options(selectinload(OrderModel.items)) # <--- Eager Loading
    .where(OrderModel.id == order_id)
)

Async Context Manager UoW

Here's how to implement async with uow: properly. Notice the rollback in case of an exception.

class AsyncUoW:
    def __init__(self, session_factory):
        self.session_factory = session_factory

    async def __aenter__(self):
        self.session = self.session_factory()
        return self

    async def __aexit__(self, exc_type, exc_val, exc_tb):
        try:
            if exc_type:
                await self.session.rollback() # Error -> rollback
            else:
                await self.session.commit()   # Success -> commit
        finally:
            await self.session.close()        # Always close
