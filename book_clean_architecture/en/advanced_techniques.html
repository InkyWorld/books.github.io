Advanced Techniques

Mapping, filtering, and async nuances.

When you transition from theory to practice, you face routine tasks: endless data transformations, complex filters, and async pitfalls. In this chapter, we will cover advanced survival techniques.

1. The "DTO Explosion" Problem

Architecture requires separate DTOs for each layer. In practice, for a single User entity, you need to create:

  • CreateUserRequest (Presentation, Pydantic)
  • CreateUserDTO (Application, Dataclass)
  • UserEntity (Domain)
  • UserResponse (Presentation, Pydantic)

This is exhausting and demotivating. Let\'s consider a pragmatic approach.

Duck Typing & Protocols

If the Application layer should not depend on Pydantic, this doesn\'t mean we can\'t pass Pydantic objects if they behave as needed. Use Protocols.

# application/protocols.py
from typing import Protocol

class UserInput(Protocol):
    username: str
    email: str

# application/use_cases.py
def register_user(data: UserInput):
    # Use Case knows nothing about Pydantic, but works with it
    print(data.username) 

# presentation/api.py
class CreateUserRequest(BaseModel): # Pydantic model
    username: str
    email: EmailStr

# Works thanks to "Duck Typing"
register_user(CreateUserRequest(username="alex", email="a@b.com"))

Automapping (Pragmatic)

For simple CRUD operations, you can use model_validate or helper methods to avoid manually mapping each field queue.

# If fields match 1-to-1
def to_domain(dto: CreateUserRequest) -> User:
    # Dangerous for complex logic, but OK for simple CRUD
    return User(**dto.model_dump())

2. Pagination and Filtering

Don\'t pass pagination parameters as function arguments. This violates the Open-Closed Principle (when adding sorting, you\'ll have to change the signature).

Generic Page[T]

Use Generic DTO for all list responses.

from typing import TypeVar, Generic, List
from dataclasses import dataclass

T = TypeVar("T")

@dataclass
class PaginationParams:
    page: int = 1
    limit: int = 20
    sort_by: str = "created_at"
    desc: bool = True

@dataclass
class Page(Generic[T]):
    items: List[T]
    total: int
    page: int
    size: int
    pages: int

Repository Implementation

# infrastructure/repo.py
async def search(self, filters: OrderFilter, pagination: PaginationParams) -> Page[Order]:
    query = select(OrderModel)
    
    # 1. Apply Filters
    if filters.status:
        query = query.filter_by(status=filters.status)

    # 2. Count Total (Advanced: use separate count query)
    # total = ... 

    # 3. Apply Pagination
    query = query.limit(pagination.limit).offset((pagination.page - 1) * pagination.limit)
    
    # 4. Result
    result = await self.session.execute(query)
    models = result.scalars().all()
    
    return Page(
        items=[self._mapper.to_domain(m) for m in models],
        total=100, # placeholder
        page=pagination.page,
        size=pagination.limit,
        pages=5
    )

Attempting to access order.items in async code without preloading will cause MissingGreenlet. Use Selectinload.

# Correct relationship loading in Async
stmt = (
    select(OrderModel)
    .options(selectinload(OrderModel.items)) # <--- Eager Loading

Async Context Manager UoW

Here\'s how to properly implement async with uow:. Pay attention to rollback in case of exception.

class AsyncUoW:
    def __init__(self, session_factory):
        self.session_factory = session_factory

    async def __aenter__(self):
        self.session = self.session_factory()
        return self

    async def __aexit__(self, exc_type, exc_val, exc_tb):
        try:
            if exc_type:
                await self.session.rollback() # Error -> rollback
            else:
                await self.session.commit()   # Success -> commit
        finally:
            await self.session.close()        # Always close
