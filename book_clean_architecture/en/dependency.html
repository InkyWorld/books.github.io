Dependency Injection (DI)

How to assemble an application from disparate classes without creating hard links.

1. The Problem: "Who Creates the Object?"

We have a `CreateOrderUseCase`. It needs an `OrderRepository`. Who should create the repository?

‚ùå Bad: Created Inside

class CreateOrderUseCase:
    def __init__(self):
        # Hard dependency! Impossible to test.
        self.repo = SqlAlchemyRepository()

‚úÖ Good: Passed from Outside

class CreateOrderUseCase:
    # Dependency is "injected"
    def __init__(self, repo: OrderRepository):
        self.repo = repo

The second option is better because the Use Case no longer knows what SQLAlchemy is. It works with an abstraction. But the question arises: Where do we create these objects?

2. Composition Root Pattern

This is the single place in the application where we create all objects and link them together. Usually, this happens when the application starts (for example, in the `main.py` file or a separate `dependencies.py` module).

[ main.py / Composition Root ] | 1. Creates DB Engine 2. Creates SessionFactory 3. Creates Repository(session) 4. Creates EmailService() 5. Creates UseCase(repo, email_service) <--- ASSEMBLY READY | V [ FastAPI Route ] ---> Calls the ready UseCase

3. Example: Manual Assembly (Pure Python)

In Python, you often don't need complex DI frameworks. Regular functions are sufficient.

# main.py or container.py

# 1. Import implementations (Infrastructure)
from infrastructure.db import create_engine, session_factory
from infrastructure.repositories import SqlAlchemyOrderRepository
from infrastructure.adapters import SmtpEmailService

# 2. Import Use Case (Application)
from application.use_cases import CreateOrderUseCase

# Global settings (can be taken from env)
DB_URL = "postgresql://user:pass@localhost/db"
engine = create_engine(DB_URL)

# DEPENDENCY FACTORY (Composition Root)
def get_create_order_use_case() -> CreateOrderUseCase:
    # a. Create DB session
    session = session_factory(bind=engine)
    
    # b. Create user adapters (Infrastructure)
    repo = SqlAlchemyOrderRepository(session)
    email_service = SmtpEmailService("smtp.gmail.com", ...)
    
    # c. Assemble Use Case (Injection)
    use_case = CreateOrderUseCase(
        repo=repo, 
        email_service=email_service
    )
    
    return use_case

4. Integration with FastAPI (Depends)

FastAPI has a built-in DI system (`Depends`) that is perfect for the Composition Root role.

from fastapi import APIRouter, Depends
# Import our factory (see above)
from .container import get_create_order_use_case

router = APIRouter()

@router.post("/orders")
def create_order_endpoint(
    data: CreateOrderDTO,
    # FastAPI calls get_create_order_use_case function, 
    # retrieves the ready UseCase and injects it into the use_case variable
    use_case: CreateOrderUseCase = Depends(get_create_order_use_case)
):
    return use_case.execute(data)
Pro tip: If you need to manage the lifecycle (e.g., close the DB session after a request), use `yield` in the dependency function.

5. Connecting Unit of Work

With UoW, the scheme is slightly more complex since the Repository is not created immediately, but inside a `with` context.

# container.py

# 1. First, create a session factory (not the session itself!)
from sqlalchemy.orm import sessionmaker
session_factory = sessionmaker(bind=engine)

def get_uow():
    # Pass session factory to UoW
    return SqlAlchemyUnitOfWork(session_factory)

def get_use_case_with_uow(uow = Depends(get_uow)):
    # Use Case now depends on UoW, not on Repository
    return CreateOrderWithUowUseCase(uow)

# application/use_cases.py
class CreateOrderWithUowUseCase:
    def __init__(self, uow: UnitOfWork):
        self.uow = uow

    def execute(self, cmd):
        with self.uow:
            order = Order(...)
            # Repository is available via UoW
            self.uow.orders.save(order) 
            # commit is automatic on exit

6. Dishka: Modern DI Framework Recommended

If you're tired of writing manual factories, and FastAPI Depends seems too tied to the web ‚Äî try Dishka.

It is a modern (2024+) IoC container that works everywhere (FastAPI, Aiogram, Typer, Pure Python) and has a very clean API.

Why Dishka?

  • Scope Control: Clear separation into Singleton, Request, Session scopes.
  • Clean API: Uses modern type hints.
  • Framework Agnostic: One container for both bot and API.
from dishka import Provider, Scope, provide, make_async_container
from dishka.integrations.fastapi import FromDishka, DishkaApp

# 1. Provider (Analogous to our container.py)
class AppProvider(Provider):
    
    @provide(scope=Scope.APP)
    def get_engine(self) -> AsyncEngine:
        return create_async_engine("sqlite+aiosqlite:///:memory:")

    @provide(scope=Scope.APP)
    def get_session_maker(self, engine: AsyncEngine) -> async_sessionmaker:
        return async_sessionmaker(engine)

    @provide(scope=Scope.REQUEST)
    async def get_session(self, factory: async_sessionmaker) -> AsyncIterable[AsyncSession]:
        async with factory() as session:
            yield session

    @provide(scope=Scope.REQUEST)
    def get_uow(self, session: AsyncSession) -> UnitOfWork:
        return SqlAlchemyUoW(session)

    @provide(scope=Scope.REQUEST)
    def get_create_order_use_case(self, uow: UnitOfWork) -> CreateOrderUseCase:
        return CreateOrderUseCase(uow)

# 2. Integration with FastAPI
@router.post("/orders")
async def create_order(
    cmd: CreateOrderCommand,
    # Dishka will find the required provider by type
    use_case: FromDishka[CreateOrderUseCase] 
):
    return await use_case.execute(cmd)
